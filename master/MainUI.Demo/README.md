# MainUI.Demo 独立仿真演示项目 — 集成说明

## 这个项目是什么

一个完全独立的 WinForms 应用 (`MainUI.Demo.exe`)，提供柴油机试验台仿真演示功能。**与主程序 (`MainUI.exe`) 物理隔离**，互不影响。

特性：
- 完全独立的进程和可执行文件
- 仅依赖 .NET Framework 4.5（无第三方库）
- 日志只写本地文件（无 MySQL 依赖）
- 顶部红色 SIM 横条 + SIMULATED 水印 — 视觉防误用
- 闭环响应（拖油门 → 转速/功率联动）
- 5 类故障注入预设 + 自动模拟上层响应
- 完整 TraceId 因果链
- 实时日志查看器 + 双击展开因果链详情

## 文件清单（10 个）

```
master/MainUI.Demo/                       ← 在 master 解决方案下新建子项目
├── MainUI.Demo.csproj                    项目文件
├── Program.cs                            程序入口
├── Core/                                 核心服务（无 UI 依赖）
│   ├── SimulatorDataBus.cs               仿真数据总线（隔离的关键）
│   ├── SimulationEngine.cs               仿真引擎（含一阶惯性、故障注入）
│   └── SimulationLogger.cs               中央日志服务（文件版）
└── UI/                                   界面层
    ├── FrmDemoMain.cs                    主窗口逻辑
    ├── FrmDemoMain.Designer.cs           主窗口 UI 设计
    ├── FrmSimLogViewer.cs                日志查看器
    ├── FrmSimLogViewer.Designer.cs       日志查看器 UI 设计
    └── FrmTraceDetail.cs                 因果链详情对话框（无 Designer，代码内构建）
```

## 集成步骤

### Step 1：在 master 解决方案中创建子项目

1. VS 打开 `master.sln`
2. 解决方案右键 → 添加 → 新建项目 → Windows 窗体应用 (.NET Framework)
3. 项目名: `MainUI.Demo`
4. 位置: `master\MainUI.Demo\`
5. .NET Framework: **4.5**（必须和主程序一致）

### Step 2：放置文件

按上面的目录结构把 10 个文件放进 `MainUI.Demo\` 目录：

- Core/ 三个 .cs 文件
- UI/ 五个 .cs 文件
- 根目录两个文件（Program.cs + .csproj）

如果新建项目时已自动生成了 `Form1.cs` 和 `Program.cs`，删除它们，用本套替换。

或者直接用本套的 `.csproj`：
1. 关闭 VS
2. 删除自动生成的项目
3. 用本套 `MainUI.Demo.csproj` 替换
4. 重新打开 `master.sln`

### Step 3：编译

VS 中选择 `MainUI.Demo` 项目右键 → 生成。

应无任何错误。如果有错误，最可能是：
- .NET 版本不匹配 → 项目属性里改成 .NET Framework 4.5
- 中文字符编码问题 → 确保所有 .cs 文件保存为 UTF-8 with BOM

### Step 4：运行

设置 `MainUI.Demo` 为启动项目，F5 运行。

也可以从 `bin\Debug\MainUI.Demo.exe` 直接双击启动 — **这是给客户演示用的方式**，独立运行不需要主程序。

## 操作流程（演示脚本）

### 5 分钟标准演示

| 步骤 | 操作 | 客户看到 |
|------|------|---------|
| 1 | 双击 `MainUI.Demo.exe` 启动 | 顶部红色 SIM 横条 + 大水印 "SIMULATED" + 12 个仪表显示 0 |
| 2 | 点 [▶ 启动仿真] | 顶部"会话""已运行""日志"开始走 |
| 3 | 30 秒内仪表逐渐爬升到稳态 | 转速 800rpm、功率 2160kW、高温水从 25℃ 慢慢升到 68℃ |
| 4 | 拖动油门到 50% | 仪表数字下降（闭环响应） |
| 5 | 点 [打开日志查看器] | 弹出日志窗口，看见各类别日志滚动 |
| 6 | 点 [● 高温水≥103℃降载] | 仪表"高温水"变红 + 主窗口报警；约 1 秒后油门自动归零，转速/功率掉下来 |
| 7 | 回到日志查看器双击红色行 | **弹出因果链详情** — 11 步从触发到处置全列出来 |
| 8 | 点 [📥 导出 CSV] | 弹出保存对话框 |

### 关键演示话术

> **"您看，仿真器每一个动作 — 什么时候开始、注入了什么值、上层逻辑用了多少毫秒识别到、最后系统状态怎么变 — 全部以毫秒精度结构化记录。双击任意一条故障日志就能拉出从触发到处置的完整因果链。"**
> 
> **"最终我们会基于这些日志自动生成《仿真器测试报告》。这份报告您可以直接交给铁路局或第三方检验机构。"**

## 与主程序的关系

| 维度 | 状态 |
|------|------|
| 进程 | 完全独立 |
| 数据流 | 完全独立（用 SimulatorDataBus，不碰 Common.AIgrp） |
| 日志 | 完全独立（写 `MainUI.Demo.exe` 同目录的 SimLog 文件夹） |
| 配置 | 完全独立 |
| MySQL | 不依赖（日志只写文件） |
| 主程序代码 | **零修改** |

把 `MainUI.Demo.exe` 删了，主程序完全感受不到。

## 工程师向开发延伸方向

如果客户认可这套方案，后续可在 Demo 项目内扩展：

1. **多工况切换** — 现在硬编码标定工况，扩展为读 GKConfig
2. **复杂故障组合** — 同时触发多个故障预设
3. **历史数据回放** — 读 `TestParaALL` JSON 文件按时间戳回放
4. **测试报告生成器** — 一键从日志生成 Word 报告
5. **多型号支持** — 不只是 Z12V240ZJ
6. **场景脚本** — 保存/加载完整试验场景

所有扩展都在 `MainUI.Demo` 项目内进行，**永远不会污染主项目**。

## 风险评估

| 风险 | 等级 | 说明 |
|------|------|------|
| 主程序代码被改 | 0 | 完全没碰 |
| 主程序数据流被影响 | 0 | 不订阅 Common.AIgrp 任何事件 |
| 客户误启动 | 低 | 独立 exe，不会被主程序自动加载 |
| 视觉混淆 | 低 | 红横条 + 水印 + 标题栏明示"仿真演示版" |

## 已知限制（POC 阶段）

- 仪表盘只有 12 个核心信号，不是完整的几十个参数
- 故障注入只做了 5 种预设
- 上层响应是仿真器内部模拟的（演示用），不是真实的 FaultService 反应
- 没有数据库持久化（只写文件 .log）
- 没有自动生成测试报告功能（需要后续扩展）

这些限制在 Demo 阶段不影响演示效果，确认方向后再补充。
